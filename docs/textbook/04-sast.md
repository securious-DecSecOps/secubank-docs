# 4화 · "음수로 송금했더니 잔액이 늘어요"

## 🎬 사건

QA 담당자가 헐레벌떡 A를 찾아왔다.

> **QA**: "이거 보세요. 송금 금액에 **-1000원**을 넣었더니… 제 잔액이 *늘었어요.*"

A는 코드를 봤다. 송금 로직이 금액의 **부호를 검증하지 않는다.** 음수를 보내면 상대 계좌에서 내 계좌로 돈이 빨려온다. 명백한 결함이다. A는 자신만만했다 — *"이런 건 SAST(정적분석)가 당연히 잡겠지. 코드를 분석하는 도구잖아."*

그는 의도된 취약점 네 개(음수송금·거래내역 IDOR·회원정보 IDOR·웹쉘 RCE)가 심어진 코드에 SonarQube를 돌렸다.

## 💡 해결책 — SonarQube(SAST), "실행하지 않고 코드를 읽는다"

**왜 SAST인가?** 코드를 *돌리지 않고도* 소스의 패턴·데이터 흐름을 분석해 약점을 찾는다. 빌드 전 가장 싸고 빠른 검사라 파이프라인 앞단에 둔다.

```bash title="devsecops-path/scripts/sonarqube-scan-services.sh (발췌)"
SONAR_PROJECT_KEY="vulnbank-msa"
sonar-scanner \
  -Dsonar.projectKey="${SONAR_PROJECT_KEY}" \
  -Dsonar.host.url="${SONAR_HOST_URL}" \      # SonarQube 서버 연결
  -Dsonar.token="${SONAR_TOKEN}"              # (Jenkins credential)
# 스캔 후 Quality Gate 통과 여부를 서버에 폴링한다
```

### 🔌 파이프라인에 어떻게 꽂히나
- **스테이지**: `stage('SonarQube SAST') { sh 'bash scripts/sonarqube-scan-services.sh' }`
- **연결**: `SONAR_HOST_URL`+`SONAR_TOKEN`(2화 파라미터)으로 *별도 SonarQube 서버*에 결과를 올린다. 다른 스캐너와 달리 SAST는 서버형 — 결과·등급·Quality Gate가 서버에 산다.
- **판정**: 스캐너가 서버에 Quality Gate 통과 여부를 폴링.

## 🔍 돌려봤더니 — 멍해진 A

결과 화면을 본 A는 굳었다. 의도된 취약점 **네 개 중 — 0개.** 음수송금도, IDOR 두 개도, 웹쉘도, *하나도* 안 잡혔다. (실측: SAST 의도취약점 recall = **0/4**)

> **A**: "코드 분석 도구가… 명백한 버그를 하나도 못 잡는다고?"

하지만 SAST가 *놀고 있던 건 아니었다.* 다른 걸 정확히 잡았다 —
- `getstatus.php`의 **TLS 인증서·호스트명 검증 비활성화** (rule `php:S4830`/`php:S5527`, CWE-295/297)
- **약한 해시·난수** (CWE-328/338)
- Security Hotspot **45건**, 전체 보안등급 **D**

그리고 가장 섬뜩한 한 줄 — **보안등급은 D인데, Quality Gate는 `PASS`.**

## ⚠️ 한계 — 여기가 이 교본의 심장이다

면접에서 *반드시* 나오는 질문이고, 공급망 "막느냐" 논쟁의 뿌리다.

- **"왜 SAST가 음수송금·IDOR을 못 잡나요?"** → SAST는 **코드의 패턴·데이터 흐름**을 본다. "금액 부호를 검증해야 한다"는 **비즈니스 규칙**도, "이 요청자가 이 계좌의 주인인가"라는 **인가(authorization)**도, 소스에는 *틀린 것처럼 안 생겼다.* 정적분석에게 음수송금 코드는 그냥 정상적인 뺄셈이다. **로직·인가·런타임 행위는 SAST의 구조적 사각이다.**
- **"그럼 SAST는 무용인가요?"** → 아니다. A02(암호·TLS)는 *정확히* 잡았다. "무용"이 아니라 **담당 레이어가 다르다.** 단일 도구로 다 막으려는 게 착각이다.
- **"등급 D인데 게이트가 PASS면 위험한 거 아닌가요?"** → 맞다. 기본 Quality Gate는 `new_security_rating`(이번에 바뀐 *신규 코드*)만 평가했다. 기존 코드의 D는 게이트를 통과시켰다. **게이트를 켰다고 안전한 게 아니라, 게이트가 *무엇을* 평가하는지가 보안이다.**

> 📓 **A의 깨달음**: 0/4는 SAST의 실패가 아니라, *"한 층으로는 안 된다"*의 증거다. 이 0/4를 메우는 게 다음 화의 DAST다.

## 🧭 시니어의 4가지 렌즈

| 렌즈 | 이 통제가 의미하는 것 |
| --- | --- |
| **기술 (Tech)** | 정적분석 = 코드 패턴·데이터플로우. 암호/TLS(A02)는 강하지만 *로직·인가·런타임*은 구조적 사각(의도취약점 0/4) |
| **규제 (Regulation)** | ISMS-P 2.7 암호화 적용(TLS/약한 해시 탐지)·2.8 개발보안 / PCI-DSS Req 6.2·6.3 / 전자금융 개발보안 |
| **정책 (Policy)** | Quality Gate 정책이 곧 보안 — "신규 코드만 평가"의 함정처럼, *게이트가 무엇을 평가하도록 설계했나*가 실제 통제 수준을 정한다 |
| **관리 (Governance)** | 등급 D·Hotspot 45는 *누가* 트리아지하나? 게이트 정책(전체 vs 신규 평가)은 *누가* 설계·승인하나? "게이트 PASS"를 곧 "안전"으로 보고하면 거버넌스 실패다 |

> 🎤 **면접 한 줄**: *"SAST가 의도취약점을 0/4로 놓친 건 SAST가 나빠서가 아니라 담당 레이어가 달라서입니다. 더 위험한 건 보안등급 D인데 게이트가 PASS였다는 것 — 게이트 정책 설계가 정탐/오탐보다 중요할 때가 있습니다."*

---

A는 인정한다 — 코드를 *읽는* 것만으로는 음수송금·IDOR을 못 잡는다. 그렇다면 방법은 하나, **코드를 실제로 실행해서 공격해보는 것.** A는 처음으로 *공격자처럼* 자기 서비스에 요청을 쏘기 시작한다.
