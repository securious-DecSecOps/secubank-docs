    5화 · "남의 거래내역이 보여요"¶

    월요일 아침, 고객센터에서 내선이 넘어왔다. 상담사의 목소리가 한 톤 낮았다.

    "고객님 한 분이… 회원정보 수정 화면에서 계좌번호를 바꿨더니, 다른 사람의 정보가 떴다고 하세요."

    A는 그 한 문장의 무게를 바로 알아챘다. IDOR(Insecure Direct Object Reference). 요청에 담긴 ID 하나만 남의 것으로 바꾸면 그 사람의 자원이 열린다. 은행에서 이건 "옆 사람 통장을 내 손으로 연다"는 말과 같다.

    등을 서늘하게 한 건 따로 있었다. 4화에서 SAST가 바로 이걸 0/4로 놓쳤다. 코드를 통째로 읽는다던 그 도구가, 이 결함 앞에서 침묵했다. 이유는 4화의 결론 그대로다 — 인가(authorization)는 있어야 할 코드의 부재다. "이 요청자가 이 자원의 주인이 맞는가"를 확인하는 줄이 없다는 게 버그인데, 정적분석은 있는 코드를 본다. 없는 것은 지적할 수 없다.

    A는 코드를 다시 노려보는 대신, 다른 결심을 했다.

    코드를 읽어서 안 보인다면, 코드를 실행해서 본다. 공격자가 하는 그대로.

    읽기와 실행 사이¶

    SAST와 DAST는 같은 앱을 보지만 서 있는 자리가 정반대다. SAST는 소스를 읽는다 — 빌드도 실행도 없이 코드의 모양에서 결함을 찾는다. DAST(Dynamic Application Security Testing)는 떠 있는 앱에 진짜 HTTP 요청을 쏜다. 서버가 실제로 어떻게 반응하는지를 본다.

    이 차이가 IDOR·음수송금 앞에서 결정적이다. "남의 ID로 요청하면 막히는가"는 코드를 아무리 노려봐도 답이 안 나온다. 그런데 실제로 남의 ID로 요청을 보내보면 1초 만에 답이 나온다 — 막히면 403이 오고, 안 막히면 남의 데이터가 200과 함께 돌아온다. 실행은 의도를 모른 채로도 진실을 말한다. SAST가 "송금액이 0보다 커야 한다"는 의도를 몰라 침묵한 그 자리를, DAST는 의도 같은 건 묻지도 않고 결과로 답한다.

    그래서 A는 자기 손으로 자기 서비스를 공격하기로 한다. 방어자가 가장 정직해지는 순간은, 공격자의 자리에 앉아볼 때다.

    공격 시퀀스 — 로그인하고, 남이 되어본다¶

    scripts/test-msa-integration.sh — 실제 공격 시퀀스 (발췌)
    LOGIN_USERNAME="j.doe"      # 나는 평범한 사용자 j.doe로 로그인한다
NEGATIVE_AMOUNT="-10"       # 송금액에 음수를 넣는다
IDOR_TARGET_ID="2"         # 그리고 '남의' 회원 id

# ① 로그인 → 토큰 획득 (이후 요청에 이 토큰을 붙인다)
curl_form "login" "/api/v1/auth/login" \
  --data-urlencode "username=${LOGIN_USERNAME}" \
  --data-urlencode "password=${LOGIN_PASSWORD}"

# ② 음수 송금 — 4화의 그 결함을, 이번엔 '요청'으로 친다
curl_form "negative-transfer" "/api/v1/transactions/transfer" \
  --data-urlencode "amount=${NEGATIVE_AMOUNT}"

# ③ IDOR — 내 토큰을 든 채로, 남의 id=2 회원정보 수정을 시도한다
curl_form "idor-user-update" "/api/v1/settings/infoupdate" \
  --data-urlencode "id=${IDOR_TARGET_ID}"

    핵심은 ③이다. A는 자기 토큰으로 로그인한 상태 그대로, 요청 body의 id만 2로 바꿔 보낸다. 서버가 인가를 제대로 한다면 "당신은 id=2의 주인이 아니다"라며 막아야 한다. 4화에서 본 그 결함 — 주인 확인 줄이 없다 — 이 사실이라면, 서버는 순순히 남의 정보를 고쳐줄 것이다. 코드를 읽어선 답하지 못했던 질문에, 요청 한 방이 답한다.

    돌려봤더니 — 0/4가 3/4로¶

    A는 AWS에 떠 있는 frontend 게이트웨이(ClusterIP)로 이 시퀀스를 쐈다. 로컬 흉내가 아니라 실제 클러스터다.

    DAST 실행 결과 — AWS 라이브 (SSM command-id f36638de…, ns secure-path-dev)
    PASS: login returned HTTP 200 / token len=212
PASS: negative transfer request accepted                    # VULN-1 음수송금
PASS: IDOR user update request accepted (user id 2)         # VULN-3 회원정보 IDOR
PASS: PHP upload request accepted: uploads/1_msa-webshell.php
PASS: uploaded PHP file executed through frontend gateway   # VULN-4 웹쉘 RCE

    여기서 PASS는 테스트가 통과했다는 뜻이 아니다 — 공격이 통과했다는 뜻이다. 음수송금이 받아들여졌고, 남의 id=2 회원정보 수정이 성공했고, 업로드한 PHP가 게이트웨이를 거쳐 실행됐다. 첫 줄의 token len=212가 특히 뼈아프다 — 인증(너 누구야)은 멀쩡히 동작하는데, 인가(네가 그걸 해도 돼?)가 비어 있는, 교과서적인 구멍이다.

    숫자로 옮기면 이렇다. SAST 0/4 → DAST 3/4. 코드를 읽어 0이던 칸을, 코드를 실행해 3으로 메웠다. 이게 "도구를 많이 붙였다"와 결정적으로 다른 지점이다 — 같은 취약점 네 개를 두 도구가 각자 보고, 한쪽의 사각을 다른 쪽이 정확히 보완한다는 걸 숫자로 보였다. 계층방어는 구호가 아니라 이 한 줄, 0/4 → 3/4다.

    왜 4가 아니라 3인가 — 정직한 한 칸¶

    나머지 하나, 거래내역 IDOR(VULN-2)은 왜 못 잡았나. 도구의 한계가 아니다. 공격 스크립트에 그 케이스를 안 넣어서다. DAST는 쏴본 곳만 안다. 거래내역 조회 엔드포인트를 시퀀스에 넣지 않았으니, 그 구멍은 열린 채로 결과에 안 잡혔다. 케이스를 추가하면 4/4가 된다. 그래서 정직하게 3/4라고 적는다.

    이 빈 한 칸이 DAST의 본질적 약점을 그대로 드러낸다. DAST는 공격 표면(엔드포인트)을 알아야 잘 쏜다. 명세(OpenAPI)나 시나리오가 빈약하면 안 쏜 곳은 영영 사각이다. 그리고 SAST와 정반대로 — 원인을 못 짚는다. "뚫렸다"는 알아도 "코드 어느 줄 때문"인지는 모른다. SAST는 줄을 알지만 못 뚫고, DAST는 뚫지만 줄을 모른다. 둘이 짝일 수밖에 없는 이유가 여기 있다.

    실무에서 한 겹 더 무겁게 다뤄야 할 것 — 어디에 쏘느냐다. DAST는 실제로 데이터를 바꾸는 요청(음수송금·정보수정)을 보낸다. 그래서 운영 환경 직격은 금물이고, 스테이징·격리 환경에 쏘는 게 원칙이다. 이 PoC가 운영이 아니라 secure-path-dev 네임스페이스를 향해 쏜 것도 그래서다.

    A가 정리한 자리들¶

    하루를 닫으며 A는 이 도구가 조직에서 어디에 서는지를 적어봤다.

    기술적으로 DAST는 실행 기반이라, SAST가 의도를 몰라 놓친 인가·비즈니스 로직 결함을 실제 요청으로 끌어낸다. 0/4를 3/4로 바꾼 그 차이가 정확히 이 지점이다. 규제의 언어로 옮기면, IDOR은 ISMS-P 2.6(접근통제)의 응용 인가 결함이고, 배포 전에 이런 점검을 두는 것 자체가 2.11(사고 예방·취약점 점검)과 PCI-DSS의 테스트 요구를 이행한다. 다만 도구를 갖춘 것과 효과를 내는 것은 다르다 — 그래서 정책의 영역이 따로 있다. "무엇을 공격해볼지"(공격 표면·시나리오)를 누가 정하고, 발견된 결함을 며칠 안에 고칠지(조치 SLA)를 누가 소유하는가. DAST의 3/4 결과와 그 증적(요청·응답 캡처)은 결국 계층방어가 작동한다는 경영 보고의 근거가 되고, 그 근거를 관리할 책임은 보안책임자에게 있다.

    도구는 구멍을 보여줄 뿐이다. 그 구멍을 언제까지 누가 막을지는 사람이 정한다 — 이게 A가 5화에서 배운, 도구 사용법보다 한 겹 위의 것이었다.

    코드도 읽어봤고(SAST), 실행도 해봤다(DAST). 그런데 그날 저녁, 배포 매니페스트를 무심코 연 A의 눈에 한 줄이 걸렸다. 컨테이너가 root로 돌고 있었다. 앱을 아무리 잘 막아도, 컨테이너 설정 자체가 열려 있다면?
