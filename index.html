    Open-source · Evidence-driven DevSecOps

    보안 도구를 나열하지 않습니다.
    실제로 탐지·판정하고, 데이터로 증명합니다.

    금융권 보안담당자의 실무 — 판단 · 차단 · 추적 · 재평가 · 증적 — 을 Kubernetes 파이프라인으로 재현한 재사용 가능한 Golden Path. 취약 워크로드를 통과시켜 각 통제의 탐지 효능을 정량 지표로 검증합니다.

    처음 오셨나요? · 3분 추천 동선

    왜 → 어떻게 → 증거 순으로 보세요

    시간이 없다면 이 세 페이지만으로 왜 필요한가 · 어떻게 막는가 · 정말 잡히는가가 정리됩니다.

    1. 1
      왜 — 왜 DevSecOps인가

      기업이 보안을 좌측 이동하는 동인과, 이 PoC가 메우는 다섯 가지 통증(도입 동인·정량 효과).

    2. 2
      어떻게 — 보안 강화 조치

      그 통증을 어떤 통제로 치환했는지(무엇을·어떻게·증거). 파이프라인 동작은 CI 보안 파이프라인.

    3. 3
      증거 — 탐지 효능

      실제 측정값(SAST 0/4 → DAST 3/4, AWS 라이브 Build #3)과 증적 체인.

    The Problem · 왜 만들었나

    "이 코드, 내보내도 됩니까?" — 매 배포의 판단을 증적으로

    도구를 많이 붙였다고 안전해지지 않습니다. 문제의식에서 출발해, 최근 공급망 위협에 맞서, 전 구간 계층방어로 답합니다.

    1. 1
      문제의식

      단일 도구엔 사각이 있고(SAST는 IDOR·로직 결함을 못 봄), 배포 1회 검사로는 내일의 신규 CVE를 못 막으며, 판단·차단·증적이 흩어지면 감사가 불가능합니다. 문제 상세 →

    2. 2
      최근 위협

      2026년 공급망 공격(axios npm 침해 · TanStack CI 침해 · Shai-Hulud 웜)은 신뢰된 의존성 · CI 인프라 · 런타임 페이로드를 노립니다. 빌드 1회 SCA로는 0-day 악성 패키지를 못 잡습니다. 공급망 방어 →

    3. 3
      우리의 대응

      source→build→image→deploy→runtime→evidence 전 구간에 통제를 계층으로 깔고, 각 통제가 무엇을 잡고 무엇을 놓치는지 데이터로 증명합니다. 보안담당자의 질문을 위협 시나리오로 재현합니다. 위협 시나리오 →

    How it works

    골든패스 흐름

    1. 1
      Push

      개발자가 GitHub에 코드를 push.

    2. 2
      CI 보안검사

      Jenkins가 SAST·secret·IaC·SBOM·이미지 CVE를 스캔하고 게이트로 판정.

    3. 3
      Harbor

      통과 이미지를 레지스트리에 push, 증적 아카이브.

    4. 4
      GitOps 배포

      ArgoCD가 k3s에 자동 동기화(Git = 단일 진실).

    5. 5
      런타임 보안

      Cilium·Falco가 실행 중 행위를 관측·차단.

    6. 6
      증적·Triage

      모든 결과가 DefectDojo로 모여 보안담당자가 triage.
